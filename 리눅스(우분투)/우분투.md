### 1. 🚀 Bash Shell 구동 파이프라인 (30p 해부)
> [!danger] 🚨 보안 엔지니어의 시선: 해커의 백도어(Backdoor) 은닉처
> 리눅스가 부팅되고 사용자가 로그인할 때, 아래의 파일들이 **'순서대로 자동 실행'**된다. 해커들은 이 파일들(`.bashrc` 등)의 맨 밑줄에 악성 스크립트를 한 줄 몰래 적어둔다. 그러면 관리자가 로그인할 때마다 악성코드가 자동으로 실행(Trigger)된다.

| 순서 | 실행 주체 및 파일 | 아키텍처 역할 (의미) |
| :---: | :--- | :--- |
| **1** | `systemd` (또는 `init`) | 리눅스의 창조주. **PID 1번**을 가지며 모든 프로세스의 부모 역할을 함. `/bin/login` 프로세스를 호출하여 로그인 화면을 띄움. |
| **2** | `/bin/login` | 사용자가 ID/PW를 치면, **`/etc/passwd`**와 **`/etc/shadow`** 장부를 뒤져서 신원을 확인(Authentication)함. |
| **3** | `/bin/bash` | 로그인이 성공하면, 해당 유저에게 할당된 기본 쉘(Bash) 프로그램을 메모리에 올림. |
| **4** | **`/etc/profile`** | 🌟 **[전역 설정]** 시스템의 **모든 사용자**에게 공통으로 적용되는 환경 변수(언어, 지역 등)를 세팅함. |
| **5** | **`/etc/bash.bashrc`** | 🌟 **[전역 설정]** 모든 사용자의 터미널 창 크기, `sudo` 실패 시 출력 메시지 등을 세팅함. |
| **6** | **`~/.profile`** | 👤 **[지역 설정]** 로그인한 **특정 사용자(나)**에게만 적용되는 PATH(명령어 경로) 환경 변수를 세팅함. |
| **7** | **`~/.bashrc`** | 👤 **[지역 설정]** 특정 사용자만의 단축어(Alias), 커스텀 함수, 터미널 색상 등을 세팅함. (가장 많이 수정하는 파일) |
| **8** | **Shell Prompt** | 모든 세팅이 끝나고 드디어 `unoh@server:~$` 프롬프트가 화면에 뜸. |

> **[핵심 요약]** `/etc/` 밑에 있는 건 **'모두에게(Global)'** 적용되고, `~/` (홈 디렉토리) 밑에 있는 숨김 파일(`.`)은 **'나에게만(Local)'** 적용된다.

### 2. ⚙️ Bash Shell 명령 처리 아키텍처 (31p 해부)
>[! abstract] `&`(쉬프트+7)를 명령어 뒤에 붙이면 명렁어를 백그라운드에서 실행한다

> [!danger] 🚨 보안 엔지니어의 시선: "명령어는 쉘이 실행하지 않는다"
> 터미널(`$`, `#`)에 명령어를 치면, 쉘(Bash)이 직접 일하는 게 아닙니다. 쉘은 자신의 유전자를 그대로 복제하여 **'자식 프로세스(Child Process)'**를 낳고(Fork), 그 자식에게 명령을 대신 수행(Exec)하게 시킵니다.

![[캡쳐 창고/리눅스(우분투) 26.webp]]
(31p의 그림)
**[순서도 완벽 해독: Foreground vs Background]**
명령어를 칠 때, 맨 뒤에 **`&` (앰퍼샌드)**를 붙이느냐 마느냐가 시스템의 운명을 가릅니다.

| 흐름 (Flow) | 아키텍처 원리 | 실무 예시 및 결과 |
| :--- | :--- | :--- |
| **NO (`&` 없음)**<br>*(Foreground)* | 쉘(부모)은 자식 프로세스가 일을 다 끝내고 죽을 때까지 **아무것도 안 하고 대기(Wait)**한다. | `sleep 100`<br>100초 동안 터미널이 멈춰서 다른 명령어를 칠 수 없음. (답답함) |
| **YES (`&` 있음)**<br>*(Background)* | 쉘(부모)은 자식에게 일을 던져놓고, **결과를 기다리지 않고 즉시 다음 명령을 받을 준비(대기 상태)**를 한다. | `sleep 100 &`<br>자식은 뒤에서 100초를 세고 있고, 나는 터미널에서 바로 `ls`나 `cd`를 칠 수 있음. |

> **[클라우드 실무 꿀팁]**
> AWS 서버에서 10시간짜리 데이터 백업 스크립트를 돌릴 때, 그냥 실행(Foreground)하고 퇴근하면? 노트북 덮는 순간 SSH가 끊기면서 백업도 같이 죽어버립니다. 반드시 명령어 뒤에 **`&`**를 붙여서 백그라운드로 던져놔야 서버가 혼자 밤새워 일합니다.

### 3. 🗂️ 리눅스 최상위,핵심 디렉토리 아키텍처 (34,35p 해부)

> [!tip] 💡 리눅스는 `C:\`, `D:\` 같은 드라이브 개념이 없다. 오직 **`/` (Root)**라는 거대한 하나의 뿌리에서 모든 가지(디렉토리)가 뻗어 나간다.

> [!tip] 💡 리눅스의 모든 것은 파일(File)이며, 디렉토리 이름은 그 파일의 '목적'을 규정한다.

| 디렉토리           | 역할 및 보안 관점의 의미                                                                                                                                   |
| :------------- | :----------------------------------------------------------------------------------------------------------------------------------------------- |
| **`/` (Root)** | **최상위 디렉토리.** 리눅스 파일 시스템의 절대적인 시작점.                                                                                                              |
| **`/boot`**    | 리눅스의 심장(Kernel)과 부팅 설정 파일이 있는 곳. 🚨 **[보안 경고]** 여기 파일 하나라도 지우면 서버는 영원히 부팅되지 않는 벽돌이 됨.                                                            |
| **`/bin`**     | **(Binary)** `ls`, `cd`, `cp` 등 모든 사용자가 쓸 수 있는 가장 기본적인 실행 파일(명령어)들이 모여 있는 곳.                                                                     |
| **`/etc`**     | **(Et Cetera)** 시스템의 **모든 환경 설정 파일(Config)**이 모여 있는 뇌. 🚨 **[보안 경고]** 해커들이 서버를 장악하면 가장 먼저 `/etc/passwd`와 `/etc/shadow`를 털러 옴.                    |
| **`/dev`**     | **(Device)** 리눅스는 하드디스크, 마우스, 키보드조차 모두 '파일'로 취급함. 물리적 장치들이 파일 형태로 매핑되어 있는 곳.                                                                     |
| **`/root`**    | **최고 관리자(root) 전용 홈 디렉토리.** 일반 유저의 홈(`/home/user`)과는 물리적으로 완벽히 격리된 신(God)의 방.                                                                    |
| **`/home`**    | 일반 사용자들의 방이 모여 있는 아파트. 계정을 만들면 `/home/unoh` 처럼 개인 공간이 할당됨.                                                                                       |
| **`/lib`**     | **(Library)** 윈도우의 `C:\Windows\System32`와 같음. 시스템과 명령어들이 돌아가기 위해 필요한 필수 공유 라이브러리(부품)들이 모여 있는 곳.                                                  |
| **`/mnt`**     | **(Mount)** 관리자가 새로운 하드디스크나 네트워크 스토리지(NFS)를 추가했을 때, **수동으로 연결(마운트)**하여 사용하는 빈 폴더.                                                                |
| **`/media`**   | **(Media)** USB나 CD-ROM을 꽂았을 때 OS가 **자동으로 연결(마운트)**해 주는 폴더.                                                                                      |
| **`/tmp`**     | **(Temporary)** 임시 파일 저장소. 재부팅하면 싹 다 날아감. 🚨 **[보안 경고]** 누구나 읽고 쓸 수 있는 `777` 권한에 `Sticky Bit(t)`가 걸려있어, 해커들이 악성코드를 다운로드하는 1순위 베이스캠프임.            |
| **`/var`**     | **(Variable)** 용량이 계속 변하는(Variable) 파일들이 모이는 곳. 🚨 **[보안 경고]** `/var/log`에는 시스템의 모든 접속 기록과 에러 로그가 남으므로, 보안 엔지니어가 매일 아침 가장 먼저 까봐야 하는 **'블랙박스'**임. |

## ⌨️ vi/vim 에디터 핵심 단축키 총정리

> [! danger] 💀 보안 엔지니어의 시선: 권한 상승 (Privilege Escalation) 취약점
>`vi`나 `vim`이 `sudo` (관리자 권한)로 실행 중일 때, 내부에서 **`:!/bin/sh`** 또는 **`:!/bin/bash`** 를 입력하면, 에디터를 빠져나와 **Root 권한의 쉘(Shell)**이 그대로 열려버린다. 
>따라서 실무 서버에서는 일반 사용자에게 `sudo vi` 권한을 함부로 부여하면 서버가 1초 만에 완벽하게 장악(Takeover)당할 수 있다. (대안: `sudoedit` 사용)

### 1. VI/VIM 모드 종류 및 전환
>[!warning] 🚨 vi의 핵심: 현재 내가 무슨 모드인지 파악하는 것이 생명이다. 길을 잃으면 무조건 `ESC`를 연타하라.

| 모드 | 설명 | 전환 방법 |
| :--- | :--- | :--- |
| **일반 모드 (Normal Mode)** | vi를 실행하면 최초로 진입하는 기본 모드 | `ESC` |
| **입력 모드 (Insert Mode)** | 텍스트를 직접 타이핑하여 입력하는 모드 | `a`, `i`, `o` |
| **비주얼 모드 (Visual Mode)** | 텍스트를 블록 지정(선택)하는 모드 | `v`, `V`, `Ctrl+v` |
| **명령 행 모드 (Command Line)** | 저장, 종료, 환경 설정 등을 입력하는 모드 | `:` |
| **검색 (Search)** | 파일 내에서 특정 문자열을 찾는 모드 | `/` 또는 `?` |

### 2. 커서 이동 (Navigation)
> [!tip] 💡 고인물은 방향키(화살표)를 쓰지 않는다. 오직 알파벳만으로 커서를 날아다니게 하라.

| 단축키 | 설명 |
| :--- | :--- |
| `gg` | 파일의 **첫 번째 줄**로 이동 |
| `G` | 파일의 **마지막 줄**로 이동 |
| `nG` 또는 `ngg` | `n`번째 줄로 이동 (예: `22G` ➡️ 22번째 줄로 이동) |
| `Ctrl+u` / `Ctrl+d` | 화면을 위(Up) / 아래(Down)로 반 페이지씩 스크롤 |
| `w` | 다음 단어의 **처음**으로 이동 (Word) |
| `e` | 현재 단어의 **끝**으로 이동 (End) |
| `f<알파벳>` | 현재 행에서 입력한 특정 알파벳 위치로 점프 (Find) |
| `^` | 현재 행의 **맨 앞**으로 이동 (공백 제외) |
| `$` | 현재 행의 **맨 끝**으로 이동 |

### 3. 텍스트 조작 (삭제/복사/붙여넣기)
> [!danger] 🚨 `dd`는 단순한 삭제가 아니라 '잘라내기(Cut)'다. 지운 내용은 클립보드에 남아 `p`로 붙여넣을 수 있다.

| 단축키 | 설명 |
| :--- | :--- |
| `dd` | 현재 행 전체 삭제 (잘라내기) |
| `D` | 커서 위치부터 그 행의 끝까지 삭제 |
| `yy` | 현재 행 전체 복사 (Yank) |
| `p` / `P` | 복사하거나 잘라낸 내용을 커서 뒤(`p`) 또는 앞(`P`)에 붙여넣기 (Paste) |
| `u` | 실행 취소, 이전 작업으로 되돌리기 (Undo) |
| `Ctrl+r` | 다시 실행, Undo 한 것을 되돌리기 (Redo) |

### 5. 명령 행 모드 (Command Line)
> [!warning] 🚨 무지성 `:wq`는 파일의 수정 시간(mtime)을 갱신하여 서버 오작동을 유발할 수 있다. 실무에서는 `:x`를 애용하라.

| 단축키 | 설명 |
| :--- | :--- |
| `:w` | 파일 저장 (Write) |
| `:q` | 저장하지 않고 종료 (Quit) |
| `:wq` 또는 `:x` | 저장하고 종료 (`:x`는 변경 사항이 있을 때만 저장하는 스마트 종료) |
| `:q!` | 변경 사항을 무시하고 **강제 종료** (설정 파일 건드리다 꼬였을 때 비상 탈출구) |
| `:set number` | 에디터 왼쪽에 줄 번호 출력 |
| `:set nonumber` | 줄 번호 숨김 |
| `:set hlsearch` | 검색 결과에 하이라이트(형광펜) 표시 |
| `:set ignorecase` | 대/소문자 구분 없이 검색 |
| `:s/pattern/replacement/` | 검색한 문자열(`pattern`)을 새로운 문자열(`replacement`)로 교체 |
>[!danger]- 🚨 `:wq` 무지성 사용의 위험성 (mtime 갱신)
> `:wq`는 변경 사항 유무와 관계없이 무조건 디스크에 Write(I/O)를 발생시켜 파일의 **수정 시간(mtime)**을 갱신한다. 이는 파일 변경을 감지하여 작동하는 자동화 스크립트(CI/CD, Make 등)를 오작동하게 만드는 주범이다.
  
  > [!tip]- 🚨 `:` 와 `:!` 의 본질적 차이
> - **`:` (Internal Command):** `vi` 에디터 자체의 내장 명령어를 실행한다. (저장, 종료, 치환 등)
> - **`:!` (External Shell Command):** 에디터를 종료하지 않고, 호스트 OS의 쉘(Bash 등)을 호출하여 외부 리눅스 명령어를 실행하는 **'Shell Escape'** 기능이다.
---
- **🛠️ 실무 엔지니어의 `:!` 활용법 (생산성 극대화)**
  에디터 작업 중 흐름을 끊지 않고 OS와 상호작용할 때 사용한다.
  1. **`:!ls -l`** : 현재 디렉토리의 파일 목록 확인 후 복귀. 
  2. **`:!ping 8.8.8.8`** : 네트워크 상태 확인 후 복귀.
  3. **`:r !date`** : (응용) 리눅스의 `date` 명령어 실행 결과x를 **현재 커서 위치의 텍스트로 그대로 삽입(Read)**한다. (스크립트 작성 시 개꿀팁)

| 옵션   | 설명 (예시 명령어 매핑)                                                           |
| :--- | :----------------------------------------------------------------------- |
| `-m` | **홈 디렉토리 생성** (명령어의 `-m` 부분. 이 옵션이 없으면 폴더가 안 생김)                         |

| `-u` | 계정의 UID 수동 설정                                                            |
| `-U` | 계정명과 동일한 이름의 그룹을 자동으로 생성                                                 |
| `-r` | **시스템 계정으로 생성** (UID 1000 미만의 서비스 구동용 계정 생성 시 사용)                        |

### 1. 권한의 뼈대 (10자리 문자열)
`ls -al`을 쳤을 때 나오는 `drwxr-xr-x` 같은 10자리 외계어의 완벽한 해부도입니다.

|   자리수    | 분류              | 기호 및 의미                               |
| :------: | :-------------- | :------------------------------------ |
|  **1**   | **파일 타입**       | `-` (일반 파일), `d` (디렉토리), `l` (심볼릭 링크) |
| **2~4**  | **Owner (소유자)** | 파일을 만든 주인이 가지는 권한 (`rwx`)             |
| **5~7**  | **Group (그룹)**  | 파일이 속한 그룹원들이 가지는 권한 (`r-x`)           |
| **8~10** | **Other (그 외)** | 주인도 아니고 그룹원도 아닌 '지나가는 행인'의 권한 (`r-x`) |

---

### 2. 8진수 연산 (Octal Mode)
컴퓨터는 `rwx`를 모릅니다. 오직 이진수(Bit)의 합으로만 권한을 계산합니다.


|        권한        | 기호  | 이진수 | 8진수 값 |
| :--------------: | :-: | :-: | :---: |
|  **Read (읽기)**   | `r` | 100 | **4** |
|  **Write (쓰기)**  | `w` | 010 | **2** |
| **Execute (실행)** | `x` | 001 | **1** |
|    **권한 없음**     | `-` | 000 | **0** |

> **[실무 계산법]** `rwxr-xr--` ➡️ (4+2+1) / (4+0+1) / (4+0+0) ➡️ **`754`**

>[! info] 원래 4자리이며 **[[우분투#4. 특수 권한 (SetUID, SetGID, Sticky Bit)|앞의 0]]** 을 생략해도 되기에 3자리로도 씀

---

### 3. 🚨 파일 vs 디렉토리 권한의 잔혹한 차이
> [!danger] 주니어들의 무덤: "파일에 쓰기(w) 권한이 있는데 왜 파일 삭제가 안 되죠?"
> **파일을 삭제하는 권한은 '파일 자체의 `w` 권한'이 아니라, 그 파일이 들어있는 '디렉토리의 `w` 권한'에 종속됩니다.**

|     권한      | 파일 (File) 에 적용될 때                                                       | 디렉토리 (Directory) 에 적용될 때                                                        |
| :---------: | :---------------------------------------------------------------------- | :------------------------------------------------------------------------------ |
| **`r` (4)** | 파일의 **내용을 읽을 수 있음** (`cat`,`vi` 읽기 모드)                                  | 디렉토리 안의 **파일 목록을 볼 수 있음** (`ls`)                                                |
| **`w` (2)** | 파일의 **내용을 수정할 수 있음** (`echo`, `vi` 저장. **`vi`에선 :wq! 명령으로 소유자만 저장 가능**) | 디렉토리 안에 **파일을 생성/삭제/이름 변경할 수 있음** (`touch`, `rm`, `mv`)                         |
| **`x` (1)** | 파일을 **프로그램처럼 실행할 수 있음** (`./script.sh`)                                 | 디렉토리 안으로 **진입할 수 있음** (`cd`). <br>*(주의: 디렉토리에 `x`가 없으면 `r`이 있어도 안의 파일에 접근 불가!)* |

---

### 4. 특수 권한 (SetUID, SetGID, Sticky Bit)
> [!warning] 보안 엔지니어의 최우선 감시 대상. 해커들이 권한 상승(Privilege Escalation)을 위해 가장 먼저 노리는 백도어(Backdoor)입니다.

>[! info] 8진수의 뒤 3자리는 **[[우분투#2. 8진수 연산 (Octal Mode)|여기로]]**

| 특수 권한          |  8진수   |     기호      | 실무 적용 사례 및 의미                                                                                           |
| :------------- | :----: | :---------: | :------------------------------------------------------------------------------------------------------ |
| **SetUID**     | `4000` | `s` (Owner) | **"실행하는 순간만 주인의 영혼을 빙의한다."**<br>예: `/usr/bin/passwd` (일반 유저가 자기 비밀번호를 바꿀 때, 실행하는 순간만 root 권한을 빌려줌)      |
| **SetGID**     | `2000` | `s` (Group) | **"실행하는 순간 그룹의 영혼을 빙의한다."**<br>디렉토리에 걸리면, 그 안에 생성되는 모든 파일이 해당 디렉토리의 그룹을 상속받음. (협업 폴더에 필수)               |
| **Sticky Bit** | `1000` | `t` (Other) | **"내 파일은 나만 지울 수 있다."**<br>예: `/tmp` 폴더. 누구나 파일을 만들 수(`w`) 있지만, 삭제는 파일을 만든 본인(또는 root)만 가능하게 멱살을 잡는 권한. |

---

### 5. 권한 제어 명령어 (`chmod`, `chown`)

#### `chmod` (Change Mode) : 방화벽 룰 변경
```bash
# 1. 8진수(Octal) 절대 지정법 (실무 표준)
chmod 755 script.sh   # Owner: rwx(7), Group: r-x(5), Other: r-x(5)
chmod 600 secret.key  # Owner: rw-(6), 나머지 접근 불가 (SSH 키 파일 필수 권한)

# 2. 심볼릭(Symbolic) 상대 지정법
chmod u+x script.sh   # User(주인)에게 eXecute(실행) 권한 추가
chmod o-r secret.txt  # Other(남)에게서 Read(읽기) 권한 박탈
```

#### `chown` (Change Owner) : 명의 이전
| 옵션 | 설명 | 보안 엔지니어의 경고 (Risk) |
| :---: | :--- | :--- |
| **`-v`** | **(Verbose)** 변경된 파일의 정보를 화면에 자세히 출력함. | 안전함. 대량의 파일 소유권 변경 시 진행 상황 모니터링용으로 필수. |
| **`-R`** | **(Recursive)** 지정한 디렉토리와 그 안의 모든 하위 파일/폴더의 소유권을 일괄 변경함. | 🚨 **[위험]** 최상위 경로(`/`나 `/etc`)에서 실수로 실행할 경우, 시스템의 모든 권한이 붕괴되어 서버가 즉사(Kernel Panic)할 수 있음. |
| **`-H`** | **(Half-Logical)** `-R`과 함께 사용 시, 명령줄에 지정된 **심볼릭 링크(바로가기)를 타고 들어가 원본 파일의 소유권까지 변경**함. | 💀 **[치명적 위험]** 폴더 내부에 시스템 핵심 경로(`/`)를 가리키는 심볼릭 링크가 존재할 경우, 의도치 않게 OS 전체의 소유권이 탈취(오염)되는 대참사 발생. |
```bash
# 구문: chown [옵션] [소유자]:[그룹] [파일/디렉토리]
sudo chown nginx:nginx /var/www/html  # 소유자와 그룹을 모두 nginx로 변경
sudo chown -R unoh:devteam /project   # -R (Recursive): 폴더 안의 모든 하위 파일까지 싹 다 명의 이전
sudo chown -v nginx:nginx /var/www/html/index.html   # -
sudo chown -R -v unoh:devteam /home/unoh/project/   # -
```
### 🧮 리눅스 아키텍처: `umask`와 권한의 탄생 (Logical Operation)

> [!danger] 🚨 파일 생성 시 권한은 어떻게 결정되는가?
> 리눅스에서 파일을 새로 만들 때마다 일일이 `chmod`를 칠 수는 없다. OS는 파일과 디렉토리가 생성될 때 **'최대 허용 권한'**에서 **'umask(마스크)'** 값을 빼서(정확히는 비트 연산하여) 기본 권한을 자동으로 부여한다.

#### 1. 최대 허용 권한 (Base Permission)
- **디렉토리 (Directory):** `777` (`rwxrwxrwx`) - 폴더는 안으로 들어가야 하므로 기본적으로 `x`가 필요함.
- **일반 파일 (File):** `666` (`rw-rw-rw-`) - 일반 파일은 보안상 기본적으로 실행(`x`) 권한을 절대 주지 않음.

#### 2. `umask`의 논리 연산 아키텍처 (118p 해부)
![[캡쳐 창고/리눅스(우분투) 19.webp]]
> [! success] `/etc/login.defs` 파일에서 기본 `UMASK` 값을 수정 가능.
- `umask`는 "이 권한만큼은 절대 주지 마라!"라고 가리는 **'차단 필터(Subtractor)'** 역할을 한다.
  단순한 뺄셈이 아니라 **NOT 연산과 AND 연산**의 결합이다.
> **공식:** `New Permission = Base Permission AND (NOT umask)`

**[실무 계산 예시: `umask`가 `0022`일 때]**
*   `0022`의 의미: Group과 Other에게 쓰기(`w`, 2) 권한을 주지 마라!

| 구분        | 디렉토리 생성 시 (`777`)       | 일반 파일 생성 시 (`666`)      |     |
| :-------- | :---------------------- | :---------------------- | --- |
| **Base**  | `777` (`rwxrwxrwx`)     | `666` (`rw-rw-rw-`)     |     |
| **umask** | `- 022` (----w--w-)     | `- 022` (----w--w-)     |     |
| **결과**    | **`755` (`rwxr-xr-x`)** | **`644` (`rw-r--r--`)** |     |

| `umask` 값  | 생성되는 디렉토리 | 생성되는 파일 | 실무 적용 시나리오 및 보안 수준                                                                                                                                  |
| :--------: | :-------: | :-----: | :-------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`0002`** |   `775`   |  `664`  | [일반 유저 기본값]<br>나와 같은 그룹(Group)원들도 파일을 수정(`w`)할 수 있음. 개발팀 공용 서버나 협업 폴더에서 주로 사용.                                                                      |
| **`0022`** |   `755`   |  `644`  | [root / 시스템 기본값]<br>나만 수정(`w`)할 수 있고, 그룹과 타인은 읽기(`r`)만 가능. 가장 표준적이고 안전한 웹 서버(Apache/Nginx) 기본 세팅.                                                   |
| **`0027`** |   `750`   |  `640`  | <mark style="background: #FF5582A6;">**[보안 강화 모드]**</mark><br>나와 그룹원까지만 읽을 수 있고, 타인(Other)은 아예 접근 불가(`0`). 사내망 DB 서버나 중요 애플리케이션 서버에 적용.             |
| **`0077`** |   `700`   |  `600`  | <mark style="background: #FF0000;">**[편집증 / 최고 보안 모드]**</mark><br>오직 나(Owner)만 읽고 쓸 수 있음. 그룹과 타인은 완벽히 차단. SSH 개인키(`id_rsa`)나 인증서가 저장되는 폴더에 강제되는 권한. |

**💻 실무 적용법**
- 일회성 적용: 터미널에 `umask 0027` 입력 (로그아웃 시 초기화)
- 영구 적용: `~/.bashrc` 또는 `/etc/profile` 맨 밑줄에 `umask 0027` 추가

## 🧠프로세스(Process)와 스레드(Thread)
### 1. 🖥️ 프로세스 (Process)와 멀티 태스킹
> **[PDF 핵심 요약]** 
> - **프로세스:** 디스크에 있던 프로그램이 메모리(RAM)에 적재되어 CPU에 의해 실행되는 상태.
>
> - **멀티 태스킹:** 하나의 CPU가 여러 프로세스를 아주 빠르게 전환하며 동시에 실행되는 것처럼 보이게 하는 기술.

**[💡 수석 아키텍트의 심층 분석: EEVDF 스케줄러의 등장]**
강도님, CPU가 프로세스를 빠르게 전환(Context Switching)할 때 "누구를 먼저, 얼마나 실행시킬 것인가?"를 결정하는 뇌를 **'CPU 스케줄러'**라고 합니다.
*   **과거 (CFS):** 지난 10년 넘게 리눅스는 CFS(Completely Fair Scheduler)라는 놈을 썼습니다. 무조건 공평하게 시간을 나누는 방식이었죠.
*   **현재 (2026년 최신 트렌드):** 강도님이 지금 실습 중인 **Ubuntu 24.04 LTS (커널 6.8 이상)**부터는 **EEVDF (Earliest Eligible Virtual Deadline First)**라는 차세대 스케줄러가 기본 탑재되었습니다. 
*   **왜 바뀌었나?:** 클라우드 환경에서는 '공평함'보다 '지연 시간(Latency) 최소화'가 생명입니다. EEVDF는 프로세스의 '마감 시간(Deadline)'을 계산해서 급한 놈부터 먼저 처리합니다. AWS나 GCP 같은 클라우드 서버의 응답 속도를 비약적으로 끌어올린 현대 리눅스의 핵심 아키텍처입니다.

### 2. 🧵 스레드 (Thread)와 메모리 공유의 마법
> **[PDF 핵심 요약]** 
> - **스레드:** 프로세스 내에서 실행되는 최소 단위. 
> - 프로세스 내의 자원(메모리, 파일)을 공유하지만, 각 스레드는 자신만의 **레지스터(Register)와 스택(Stack)**을 독립적으로 가짐.

**[💡 수석 아키텍트의 심층 분석: 왜 스택만 따로 가질까?]**
이게 클라우드 엔지니어 기술 면접 단골 질문입니다. 프로세스가 하나의 '공장'이라면, 스레드는 그 안에서 일하는 '일꾼'입니다.
*   **공유하는 것 (Code, Data, Heap):** 일꾼들은 공장의 기계(Code), 공용 창고(Data), 동적 자재(Heap)를 같이 씁니다. 그래서 통신 속도가 미친 듯이 빠르고 메모리를 아낍니다.
*   **독립적인 것 (Stack, Register):** 하지만 일꾼마다 자기가 지금 무슨 일을 하고 있었는지 기억할 '개인 수첩(Register)'과 '개인 작업대(Stack)'는 따로 있어야 합니다. 안 그러면 일이 꼬이니까요.

### 3. 🚨 멀티 스레딩 (Multithreading)과 보안의 치명적 약점
> **[PDF 핵심 요약]** 
> - 하나의 프로세스 내에서 여러 스레드를 동시에 실행하여 성능을 향상시키는 기술. 
> - 자원을 공유하므로 **'데이터의 일관성(Data Consistency)'**을 유지해야 함.

**[💡 수석 아키텍트의 심층 분석: 레이스 컨디션 (Race Condition)]**
강도님, PDF에 적힌 저 **'데이터의 일관성'**이라는 단어 하나가 무너지면 클라우드 보안 전체가 박살 납니다.
*   **보안 취약점의 근원:** 두 개의 스레드가 동시에 하나의 공유 자원(예: 은행 잔고, 비밀번호 파일)에 접근해서 수정하려고 할 때 발생하는 충돌을 **레이스 컨디션(경쟁 상태)**이라고 합니다.
*   **2025~2026년 최신 해킹 트렌드:** 최근 발표된 **CVE-2025-14017 (libcurl 다중 스레드 취약점)**이나 **CVE-2025-26649 (Windows Secure Channel 취약점)** 모두 이 레이스 컨디션을 악용한 사례입니다. 해커들은 스레드들이 자원을 공유하는 그 0.0001초의 틈(타이밍)을 파고들어, 권한 검사를 우회(Bypass)하거나 시스템을 장악합니다.
*   **해결책: ** 개발자와 엔지니어는 스레드가 공유 자원에 접근할 때 남들이 못 건드리게 자물쇠를 채우는 **뮤텍스(Mutex)**나 **세마포어(Semaphore)** 아키텍처를 반드시 설계해야 합니다.
### 🧬프로세스의 종류 3개와 권한 상속 (UID)
#### 1. 👑 프로세스 권한의 절대 규칙: RUID vs EUID
> **[PDF 핵심 요약]** 
> - 프로세스는 프로그램을 실행한 사용자의 UID를 할당받아 소유권을 갖는다.
> - **예외:** SetUID가 설정된 파일은 실행자의 UID가 아닌 **'파일 소유자의 UID'**로 할당된다.

**[💡 수석 아키텍트의 심층 분석: 해커들의 성배, SetUID]**
강도님, 어제 우리가 `/usr/bin/passwd` 파일에서 봤던 그 뻘건 `s` (SetUID) 기억하십니까? 리눅스 커널은 프로세스의 권한을 두 가지로 쪼개서 관리합니다.
*   **RUID (Real UID):** "이 프로그램을 진짜로 실행시킨 놈은 누구인가?" (예: 강도님, UID 1000)
*   **EUID (Effective UID):** "이 프로세스가 시스템 자원을 건드릴 때, **실질적으로 행사하는 권한**은 누구의 것인가?"
*   **대참사의 원리:** 일반적인 프로그램은 RUID = EUID입니다. 하지만 **SetUID가 걸린 프로그램은 실행되는 순간 EUID가 파일의 소유자(root, UID 0)로 변신**합니다. 해커들은 서버에 침투하면 가장 먼저 이 EUID를 0으로 조작할 수 있는 취약한 SetUID 파일을 미친 듯이 찾아다닙니다. (이게 권한 상승 해킹의 근본입니다.)

#### 2. 🌍 프로세스의 3가지 생태계 (Interactive, Batch, Daemon)

#### A. 대화형 프로세스 (Interactive Process)
> **[PDF 요약]** 터미널과 세션을 통해 사용자와 정보를 주고받으며 실행됨.
> (예: SSH 접속 후 치는 명령어들)

*   **💡 아키텍트의 시선 (SIGHUP의 공포):** 대화형 프로세스는 강도님의 모니터(TTY/PTY)에 탯줄이 연결되어 있습니다. 만약 강도님이 AWS 서버에서 10시간짜리 스크립트를 돌리다가 **노트북을 덮어서 SSH 세션이 끊기면?** 리눅스는 해당 터미널에 연결된 모든 대화형 프로세스에게 `SIGHUP(Signal Hang Up)`이라는 사약(Kill Signal)을 내려서 싹 다 죽여버립니다. 
*   **해결책:** 이를 막으려면 명령어 앞에 `nohup`을 붙이거나, `tmux` 같은 가상 터미널을 써서 프로세스의 탯줄을 터미널에서 분리해야 합니다.

#### B. 배치 프로세스 (Batch Process)
> **[PDF 요약]** 일련의 작업을 몰아서 특정 시각에 실행.
> 터미널 입출력이 없음. 시스템 사용률이 낮을 때 유용.

*   **💡 아키텍트의 시선 (Cronjob 백도어):** 실무에서는 `cron`이나 `systemd.timer`를 이용해 새벽 3시에 백업을 돌리는 용도로 씁니다.
*   **보안 취약점:** 만약 root(관리자)가 매일 새벽 3시에 실행하는 배치 스크립트 파일의 권한을 `777`로 실수해 뒀다면? 해커는 그 스크립트 내용만 악성코드로 쓱 바꿔치기합니다. 그러면 새벽 3시마다 서버가 스스로 해커에게 root 권한을 상납하는 **'완벽한 자동화 백도어'**가 완성됩니다.

#### C. 데몬 (Daemon)
> **[PDF 요약]** 백그라운드에서 계속 실행되는 프로세스.
> 웹 서버(Nginx), DB 서버(MariaDB) 등이 이에 해당.

*   **💡 아키텍트의 시선 (고아 프로세스의 승리):** 데몬(Daemon)은 그리스 신화의 '수호신'에서 따온 이름입니다. 데몬은 터미널의 지배를 받지 않기 위해, 실행되자마자 스스로 부모(터미널)와의 연결을 끊고 **'고아(Orphan)'**가 됩니다. 
*   그러면 리눅스의 창조주인 **PID 1번(`systemd`)**이 이 고아를 입양해서 백그라운드에서 영원히 살게 해줍니다. 나중에 `ps -ef` 명령어를 쳤을 때 TTY(터미널) 칸에 `?` (물음표)가 찍혀있는 놈들이 바로 이 위대한 데몬들입니다.
# 📦APT (Advanced Package Tool) 패키지 관리
## 1. 🧠 APT의 본질: "의존성 지옥(Dependency Hell)의 구원자"
리눅스에서 프로그램(패키지)을 하나 깔려면, 그 프로그램이 작동하기 위해 필요한 수십 개의 '부품(라이브러리)'들이 먼저 설치되어 있어야 합니다. 과거에는 이걸 사람이 일일이 찾아서 깔아야 하는 '의존성 지옥'이 있었습니다.
- **APT의 역할:** APT는 단순한 다운로드 프로그램이 아닙니다. 강도님이 `apt install`을 치면, **"이 프로그램이 돌아가려면 A, B, C 부품이 필요하네? 내가 알아서 순서대로 다 다운받아서 조립해 줄게!"** 하고 의존성을 완벽하게 해결해 주는 **'자동화 조립 엔진'**입니다.

## 2. 🌐 미러 사이트 (Mirror Site)란 무엇인가?
> **[아키텍트의 시선: 리눅스판 CDN (Content Delivery Network)]**
> 전 세계 수천만 대의 우분투 서버가 미국 본사 서버 하나에서만 다운로드를 받으면 서버가 터지겠죠? 그래서 우분투는 전 세계 각국에 **'본사 서버와 똑같이 복사해 둔(Mirroring) 쌍둥이 서버'**들을 만들어 두었습니다.
> - **한국 미러 서버:** `kr.archive.ubuntu.com` (카카오 등에서 스폰서링하여 운영)
> - **효과:** 내 서버와 가장 가까운 물리적 위치에서 다운로드하므로 속도(Latency)가 미친 듯이 빠릅니다.

### 🔍 세부 명령어 아키텍처 및 옵션

#### 1. `apt update` (카탈로그 동기화)
> [!danger] 🚨 주니어의 착각: "`update` 치면 프로그램이 최신 버전으로 깔리는 거 아냐?"
> **절대 아닙니다.** `update`는 프로그램을 설치하는 게 아니라, 미러 사이트에 접속해서 **"요즘 새로 나온 프로그램 목록(카탈로그) 좀 줘봐!"** 하고 내 서버의 DB(목록)만 최신화하는 작업입니다. (실제 설치/업그레이드는 `apt upgrade`나 `install`을 쳐야 함)

- **참조 경로:** `/etc/apt/sources.list.d/` (여기에 미러 사이트 주소들이 적혀 있음)
#### 2. `apt install` (패키지 설치)
미러 사이트에서 바이너리(실행 파일)와 라이브러리를 가져와 시스템의 각 디렉토리에 뿌려 넣습니다.
- **설치 경로:** 실행 파일(`/usr/bin`), 라이브러리(`/usr/lib`), 설정 파일(`/etc`)
- **로그 위치:** `/var/log/apt/history.log` (설치 및 제거 로그)
		  `/var/log/apt/term.log` (설치 과정 중 출력된 로그)

| 옵션             | 설명                                   | 실무 팁                       |
| :------------- | :----------------------------------- | :------------------------- |
| `-y`           | 설치 중 묻는 `[Y/n]` 질문에 무조건 `yes`로 자동 응답 | 쉘 스크립트로 서버 자동화 세팅할 때 필수 옵션 |
| `--reinstall`  | 이미 설치된 패키지를 강제로 덮어씌워 재설치             | 파일이 꼬였거나 깨졌을 때 복구용         |
| `--fix-broken` | 꼬여버린 의존성을 강제로 수정하며 설치                | 설치하다 에러 났을 때 최후의 보루        |

#### 3. `apt remove` vs `apt purge` (삭제)
> [!warning] ⚠️ 설정 파일을 남길 것인가, 소각할 것인가?

| 명령어/옵션                | 설명                                                          | 아키텍처 의미                                                                                                            |
| :-------------------- | :---------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------- |
| `apt remove`          | 프로그램만 지우고 **설정 파일(`/etc`)은 유지**                             | 나중에 다시 깔았을 때 기존 세팅을 그대로 쓰기 위함                                                                                      |
| `apt purge`           | 프로그램 + 설정 파일까지 **완벽하게 영구 소각**                               | 찌꺼기 하나 남기지 않는 완벽한 초기화 (Scorched Earth)<br>패키지 제거: sudo apt purge package-name                                      |
| `apt clean`           | **시스템 캐시 정리**                                               | `/var/cache/apt/archives/`에 쌓인 다운로드 설치 파일(`.deb`)들을 싹 다 지워서 디스크 용량을 확보함. (주기적으로 안 쳐주면 디스크 꽉 참)                     |
| `-s` (또는 `--dry-run`) | <mark style="background: #FF0000;">**모의 실행 (시뮬레이션)**</mark> | <mark style="background: #FF0000;">🚨 **[실무 필수]**</mark> 실제로 지우지 않고, "이거 지우면 시스템에 무슨 일이 벌어질까?"를 미리 보여줌. (대형 사고 방지) |
| `-q`                  | **출력 메시지 최소화 (Quiet)**                                      | 설치할 때 화면에 주르륵 올라오는 쓸데없는 진행 로그를 숨김. (쉘 스크립트 짤 때 화면 오염 방지용)                                                          |
| `--fix-broken`        | 꼬여버린 의존성을 강제로 수정하며 설치                                       | 설치하다 에러 났을 때 최후의 보루                                                                                                |

#### 4. `apt search` / `list` / `show` (조회)
패키지를 설치하기 전, 해당 패키지의 정체와 의존성을 파악하는 정찰(Reconnaissance) 명령어입니다.

- `apt search [키워드]`: 이름이나 설명에 키워드가 들어간 패키지 찾기 (예: `apt search nginx`)
- `apt list --installed`: 내 서버에 깔린 모든 패키지 쫙 뽑아보기
	- - `apt list --upgradable`: 현재 내 서버에 깔린 프로그램 중 **'새로운 버전(업데이트)이 나와 있는 패키지 목록'**만 쫙 뽑아보기. (보안 패치 대상 확인용)
- **`apt show [패키지명]`**: 패키지의 상세 정보 확인. (특히 아래의 **의존성 관계**를 파악할 때 씀)

| 의존성 항목           | 역할 (의미)                                              |
| :--------------- | :--------------------------------------------------- |
| **Pre-Depends ** | **[필수]** 프로그램 설치 시점에 있어야 함. (무조건 깔기 전에 깔림)           |
| **Depends**      | **[필수]** 이거 없으면 프로그램 아예 안 돌아감. (무조건 같이 깔림)           |
| **Recommends**   | **[권장]** 필수는 아니지만, 이거 없으면 핵심 기능 몇 개가 안 됨. (보통 같이 깔림) |
| **Suggests **    | **[선택]** 있으면 좋고 없어도 그만인 부가 기능.                       |
| **Conflicts**    | **[충돌]** 이 패키지랑 같이 깔려 있으면 시스템 터짐. (먼저 지워야 함)         |
| **Replaces**     | [대체] 기존 구형 패키지 갈아끼움. (기존 꺼 자동 삭제)                    |
# 🌐네트워크 관리 및 트러블슈팅 도구 (L3~L7)
> [!danger] 🚨 보안 엔지니어의 시선: 왜 이 명령어들이 생명인가?
> 서버가 해킹당하면 해커는 반드시 외부와 통신하기 위한 **'숨겨진 포트(Port)'**를 열고 악성 **'프로세스(Process)'**를 백그라운드에 띄운다. 방화벽(ACL)을 짜기 전에, 현재 내 서버의 어느 포트가 열려있고 어떤 프로세스가 그 포트를 물고 있는지(`ss`, `lsof`) 0.1초 만에 발라내야만 서버를 방어할 수 있다.

### 🔍 세부 명령어 아키텍처 및 실무 적용

#### 1. `ip addr` (인터페이스 확인)
> [!warning] ⚠️ `ipconfig`는 잊어라. 현대 리눅스의 표준은 `ip` 명령어다.

- **역할:** 서버에 꽂힌 랜선(인터페이스)의 상태와 할당된 IP 주소를 확인한다.
- **출력 해부:**
  - `lo` (Loopback): 자기 자신을 가리키는 가상 랜선 (`127.0.0.1`).
  - `ens160` (또는 `eth0`): 실제 꽂혀있는 물리적/가상 랜선 이름.
  - `inet 172.16.151.130/24`: 할당된 IPv4 주소와 서브넷 마스크(CIDR).

#### 2. `dhclient` (DHCP 임대 관리)
강도님이 패킷트레이서에서 세팅했던 그 DHCP 서버에게 IP를 구걸(?)하는 클라이언트 명령어입니다.

| 명령어                        | 설명                        | 아키텍처 의미                                                                               |
| :------------------------- | :------------------------ | :------------------------------------------------------------------------------------ |
| `sudo dhclient -r [인터페이스]` | DHCP 서비스 **해제 (Release)** | "나 이 IP 안 쓸래!" 하고 라우터에게 IP를 반납함. (인터넷 끊김)                                             |
| `sudo dhclient -v [인터페이스]` | DHCP 서비스 **요청 (Renew)**   | DORA 프로세스(Discover-Offer-Request-Ack)를 다시 태워서 새로운 IP를 받아옴. (`-v`는 진행 과정을 화면에 자세히 보여줌) |

#### 3. `ss` (소켓 통계 및 포트 스캔) - 🌟 [보안 핵심]
> [!success] 💡 실무 엔지니어의 숨쉬기 운동: `ss -tuln`
> 서버에 접속하면 무조건 이 명령어부터 치고 시작한다. 내 서버에 무슨 문(Port)이 열려있는지 확인하는 절대 공식.

| 옵션   | 의미                                                                  |
| :--- | :------------------------------------------------------------------ |
| `-t` | **TCP** 포트                                                          |
| `-u` | **UDP** 포트                                                          |
| `-l` | **Listening** (현재 외부의 접속을 기다리며 열려있는) 포트                             |
| `-n` | **Numeric** (포트 번호를 `http`, `ssh` 같은 글자가 아니라 `80`, `22` 같은 숫자로 보여줘) |

#### 4. `lsof` (열린 파일 및 포트 추적)
> [!danger] 🚨 리눅스의 철학: "모든 것은 파일이다 (Everything is a file)"
> 리눅스는 네트워크 소켓(포트)조차도 '파일'로 취급한다. `ss`로 이상한 포트가 열린 걸 발견했다면, `lsof`로 그 포트를 물고 있는 프로세스(PID)를 찾아내서 목을 쳐야(`kill`) 한다.

- **실무 명령어:** `sudo lsof -i :22`
- **해석:** "현재 **22번 포트(SSH)**를 쥐고 흔드는 프로세스가 누구인지 당장 튀어나와라!" (결과창의 `PID`를 확인하여 프로세스를 추적함)

#### 5. `mtr` (실시간 라우팅 추적)
`ping` (살아있는지 확인) + `traceroute` (어느 라우터를 거쳐가는지 확인)를 합쳐놓은 궁극의 네트워크 진단 툴입니다.
- **실무 명령어:** `mtr www.google.com`
- **효과:** 구글 서버까지 가는 길목에 있는 모든 라우터들의 IP가 뜨며, **어느 구간의 라우터에서 패킷 손실(Loss%)이 발생하고 핑이 튀는지 실시간으로 모니터링**할 수 있습니다. (네트워크 탓인지 서버 탓인지 증명할 때 씀)

#### 6. `curl` vs `wget` (웹 통신)
GUI 브라우저(크롬)가 없는 리눅스 서버에서 외부의 데이터를 가져오는 두 가지 무기입니다.

| 명령어        | 아키텍처 목적                                                                                            | 실무 활용 예시                                                        |
| :--------- | :------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------- |
| **`curl`** | **[API 통신 및 데이터 확인]**<br>HTTP 요청(GET, POST 등 HTTP Method 지원)을 보내고 결과 텍스트를 화면에 바로 뿌려줌. 단일 파일 다운 가능. | `curl -I www.google.com` (웹 서버의 헤더 정보만 빠르게 긁어올 때)               |
