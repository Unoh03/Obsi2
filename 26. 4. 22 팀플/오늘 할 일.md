## 7. 피해야 할 IP 대역


# 피해야 할 가능성이 높은 대역

- 192.168.122.0/24
  - GNS3 NAT 노드 기본 대역

- 172.16.253.0/24
  - GNS3 remote server VPN 관련 대역

- 172.17.0.0/16
  - Docker 기본 docker0 대역

- 192.168.33.0/24
  - 학원 유선 LAN underlay로 보이므로 GNS 내부망으로 쓰지 않기

- VMware가 자동 생성한 VMnet 대역
  - 충돌 가능성 있으므로 확인 후 사용


GNS3 공식 문서는 NAT 노드가 기본적으로 `192.168.122.0/24`를 사용하고, remote server VPN에는 `172.16.253.0/24`, Docker 기본에는 `172.17.0.0/16`, VMware VMnet은 `192.168.x.x/24` 대역을 자동 생성할 수 있다고 설명합니다. VMware VM을 GNS3에 붙일 때는 host-only VMnet을 쓰고 DHCP 서버를 끄라고 안내합니다. ([GNS3 Documentation](https://docs.gns3.com/docs/using-gns3/advanced/special-ip-addresses?utm_source=chatgpt.com "Special IP addresses in GNS3 | GNS3 Documentation"))

## 12. 보고서에 넣을 구조 변경 문구


# 아키텍처 변경 근거

초기 설계는 Client, DNS, APP, DB, MGMT/LOG의 5개 논리 존으로 구성하였다. 이 구조는 기능별 역할과 보안 경계를 명확히 나눌 수 있다는 장점이 있었다.

그러나 실제 구축 환경은 3대 노트북과 GNS3/VMware 기반으로 제한되어 있었다. 5개 존을 물리적으로 그대로 분산 구현할 경우, 노트북 간 연결, VMware 가상망, GNS3 서버 간 통신, 라우팅, 방화벽 문제 등 장애 지점이 지나치게 증가하였다.

이에 따라 논리 설계는 5존으로 유지하되, 실제 구현은 3대 노트북 기준의 3라우터 구조로 재설계하였다.

- A 노트북: EDGE/DNS 영역
- B 노트북: APP 영역
- C 노트북: DATA/MGMT 영역

이 구조는 물리 자원 제약을 반영하면서도, APP과 DB의 분리, DB망과 관리망의 분리 등 핵심 보안 경계를 유지할 수 있다. 또한 장애 분석 범위가 줄어들어 제한된 프로젝트 기간 안에서 실제 구축과 테스트를 완료하기에 적합하다.
