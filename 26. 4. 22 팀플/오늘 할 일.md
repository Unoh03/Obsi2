맞습니다. **기존 체크리스트는 Remote Server/Multi-server 방식이 섞여 있어서 지금 수업 방향과 안 맞습니다.**  
아래는 **UDP Tunnel 방식 기준**으로 다시 정리한 Obsidian용 체크리스트입니다.

강사님 안내서 기준으로 오늘 제출해야 하는 `IT 서비스 기획`과 `아키텍처 설계`는 보고서의 핵심 구성이고, 프로젝트 목표에도 GNS3/VMware 실제 구축과 테스트/개선점 도출이 포함됩니다.  
또 GNS3 공식 문서상 장비 간 링크 통신에는 UDP 포트가 사용될 수 있고, 방화벽이 이를 막으면 링크가 동작하지 않을 수 있습니다. ([GNS3 Documentation](https://docs.gns3.com/docs/troubleshooting-faq/troubleshoot-gns3/?utm_source=chatgpt.com "Troubleshoot GNS3 | GNS3 Documentation"))

## 1. 오늘의 기본 전략

### 핵심 원칙

오늘은 `GNS3 Remote Server / Multi-server` 방식이 아니라 **UDP Tunnel 방식**으로 진행한다.

즉, 각 PC는 자기 GNS3를 로컬에서 실행한다.

```text
A-PC GNS3          B-PC GNS3          C-PC GNS3
   |                  |                  |
 A-R  <==UDP==>     B-R  <==UDP==>     C-R
````

### 역할 구분

|구분|의미|
|---|---|
|학원 유선망 `192.168.33.0/24`|UDP Tunnel을 운반하는 바깥망, 즉 underlay|
|GNS 라우터 간 IP `10.255.x.x`|터널 안쪽에서 라우터끼리 통신하는 가상 링크망|
|A/B/C GNS 프로젝트|각 PC에서 따로 열리는 로컬 프로젝트|
|UDP Tunnel|서로 다른 PC의 GNS 장비를 이어주는 가상 케이블|

```text

```

## 12. 실패 시 트러블슈팅

### 1단계: PC 간 underlay 확인

-  A/B/C 유선 IP가 `192.168.33.x`인지 확인
    
-  PC끼리 ping 되는지 확인
    
-  Wi-Fi 꺼져 있는지 확인
    
-  C-PC IP가 최신인지 확인
    

### 2단계: UDP 포트 확인

-  A/B 포트 쌍이 서로 반대로 맞는지 확인
    
-  B/C 포트 쌍이 서로 반대로 맞는지 확인
    
-  포트 번호가 팀 내 다른 조와 겹치지 않는지 확인
    
-  방화벽이 UDP 포트를 막지 않는지 확인
    

### 3단계: GNS 연결 확인

-  UDP Tunnel 노드가 라우터 인터페이스에 연결됐는지 확인
    
-  라우터 인터페이스가 `up/up`인지 확인
    
-  라우터 IP와 마스크가 맞는지 확인
    
-  A/B 양쪽 Local/Remote port가 서로 교차되어 있는지 확인
    

### 4단계: IOS 확인

명령어:

```ios
show ip interface brief
```

확인:

```text
FastEthernet0/0    10.255.12.1    up    up
FastEthernet0/1    ...            up    up
```

### 5단계: 라우팅 확인

명령어:

```ios
show ip route
```

확인:

-  A-R에 `10.255.23.0/30` 경로가 있는가
    
-  C-R에 `10.255.12.0/30` 경로가 있는가
    
-  VPCS 테스트 시 LAN 대역 경로가 있는가

## 13. 성공/실패 판정 기준

### 성공

-  A/B PC 간 ping 성공
    
-  B/C PC 간 ping 성공
    
-  A-R ↔ B-R ping 성공
    
-  B-R ↔ C-R ping 성공
    
-  A-R ↔ C-R ping 성공
    
-  가능하면 A-VPCS ↔ C-VPCS ping 성공
    

### 조건부 성공

-  A/B UDP Tunnel만 성공
    
-  B/C는 방화벽 또는 C-PC IP 문제로 지연
    
-  오늘은 2PC 기준으로 구축 시작 가능
    

### 실패

-  PC 간 ping 불안정
    
-  UDP 포트 열어도 라우터 간 ping 불가
    
-  오전 내 원인 분리 실패
    

### 실패 시 결정

-  3PC 분산 구현은 오늘 본 프로젝트에서 제외
    
-  1PC 또는 2PC 중심 구조로 축소
    
-  3PC 분산은 보고서의 개선점/확장안으로 작성

## 15. 보고서에 넣을 구조 변경 문구

초기 설계는 Client, DNS, APP, DB, MGMT/LOG의 5개 논리 존으로 구성하였다. 이 구조는 기능별 역할과 보안 경계를 명확히 나눌 수 있다는 장점이 있었다.

그러나 실제 구축 환경은 3대 노트북과 GNS3/VMware 기반으로 제한되어 있었다. 5개 존을 물리적으로 그대로 분산 구현할 경우, 노트북 간 연결, VMware 가상망, GNS3 연결, 라우팅, 방화벽 문제 등 장애 지점이 증가하였다.

이에 따라 논리 설계는 5존으로 유지하되, 실제 구현은 3대 노트북 기준의 3라우터 구조로 재설계하였다.

- A 노트북: EDGE/DNS 영역
    
- B 노트북: APP 영역
    
- C 노트북: DATA/MGMT 영역
    

각 PC는 자기 GNS3 환경에서 로컬 장비를 실행하며, PC 간 라우터 연결은 UDP Tunnel을 활용하여 구성한다. 이 구조는 물리 자원 제약을 반영하면서도 APP과 DB의 분리, DB망과 관리망의 분리 등 핵심 보안 경계를 유지할 수 있다. 또한 장애 분석 범위가 줄어들어 제한된 프로젝트 기간 안에서 실제 구축과 테스트를 완료하기에 적합하다.

## 17. 팀원에게 말할 요약

오늘은 두 갈래로 동시에 진행한다.

1. 문서팀은 18시 제출용 IT 서비스 기획과 아키텍처 설계를 작성한다.
    
2. 네트워크 담당은 A/B/C PC에서 UDP Tunnel 방식으로 GNS 라우터 간 통신이 가능한지 검증한다.
    

핵심은 5존을 버리는 것이 아니다.

5존은 논리 설계로 유지하고, 실제 구현만 3대 노트북 기준으로 접어서 배치한다.

분산 연결은 Remote Server가 아니라 UDP Tunnel 방식으로 진행한다.

성공하면 3PC/3Router 구조로 간다.

실패하면 1PC 또는 2PC 중심 구조로 축소하고, 3PC UDP Tunnel 구조는 개선점/확장안으로 보고서에 넣는다.
